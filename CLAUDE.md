# CLAUDE.md

このファイルは、このリポジトリ（MiraStudy）で作業するすべてのAIエージェント（Claude Code、および作業を依頼された他のAIツール）が、作業を始める前に必ず読むルールブックです。

**Claude Codeを使っている場合、このファイルは自動的に読み込まれます。**
**Claude Code以外のAIエージェントに作業を依頼する場合は、最初のメッセージで「まずこのCLAUDE.mdを読んでから作業してください」と必ず伝えてください。**

詳しい理由や背景は `MiraStudy_仕様書_v4.4.md` の11章に書いてあります。ここには実務で使う要点だけをまとめます。

## このプロジェクトについて

- 子供向けの学習アプリ「MiraStudy」。**React（画面）＋ Firebase / Firestore（データ保管庫）＋（将来）Gemini API** で作る。
- **開発の“進め方”は `dev-orchestration-template`（ひな形）に沿う。** このひな形は言語非依存で、アプリ本体は TypeScript / React で作りつつ、「計画→実装→三重監査→リリース」というチーム開発の流儀・CI（自動チェック）・エージェント構成だけを取り入れる。
- 開発者（英理究さん）はプログラミング未経験。**コード中のコメントや、作業報告は日本語で、専門用語には簡単な説明を添えること。**
- 詳しい仕様は `MiraStudy_仕様書_v4.4.md` を参照。作業内容が仕様書と食い違う場合は、仕様書を優先し、勝手に仕様を変更しない。仕様を変えたほうがよいと判断した場合は、まず提案してから開発者の確認を取ること。

## コードの書き方（必ず守ること）

| 項目 | ルール |
|------|--------|
| 使用言語 | TypeScript（JavaScriptではなく） |
| コンポーネント | Reactの関数コンポーネント＋Hooks。クラス形式は使わない |
| ファイル名 | 画面部品：`PascalCase.tsx`（例：`QuestionCard.tsx`）／それ以外：`camelCase.ts`（例：`generateProblem.ts`） |
| 変数・関数名 | `camelCase`。定数は `UPPER_SNAKE_CASE` |
| フォルダ構成 | `src/components`（画面部品）／`src/features`（機能ごと）／`src/lib`（共通処理）／`src/types`（型定義） |
| コメント | 複雑な処理には日本語で「なぜこう書いたか」を一言添える |
| 整形・チェック | Prettier（自動整形）と ESLint（自動チェック）の設定に従う（保存時に自動整形されるようにしている） |
| 型チェック | `tsc --noEmit`（型の矛盾チェック）を通すこと |
| コミットメッセージ | Conventional Commits：`feat: `／`fix: `／`docs: `／`refactor: `／`test: `／`chore: ` のいずれかで始める |
| テスト | 答えを計算するロジック（テンプレート）には、必ず簡単な自動テスト（Vitest など）を添える |

## 作業の進め方（dev-orchestration-template のワークフロー）

このリポジトリは、**Orchestrator（司令塔）が専門エージェントに仕事を割り振る**やり方を土台にしています。1つのタスクは次の流れで進めます（詳しくは `docs/orchestration.md`／`agents/` を参照）。

```
計画（docs/plan.md の Next を確認）
  → 実装（Implementer）
  → テスト作成（Test Engineer）
  → 三重監査（Spec 仕様 / Security セキュリティ / Reliability 信頼性）
  → 修正ループ（「必ず直すべき指摘」がゼロになるまで繰り返す）
  → リリース判定（Release Manager が受入条件をチェックしてマージ可否を判断）
```

基本ルールは次のとおりです。

1. **`main` ブランチに直接コミットしない。** 必ず新しいブランチを作って作業する。
2. **このファイルや `docs/` の規約からズレたコードを見つけても、勝手に上書きしない。** まず差分（変更前後の違い）を確認し、意図的な変更か・単なるクセのズレかを判断する。判断がつかない場合は、作業報告の中で開発者に確認を求める。
3. **変更内容は必ずコミットとして記録する。** 「何を・なぜ変えたか」がコミットメッセージから分かるようにする。
4. **データ構造の変更、命名規則の変更など、リポジトリ全体に影響する大きな変更は、直接 `main` に反映せず、必ずPull Request（PR）を作成する。** 開発者が内容を確認し、マージ（取り込み）の判断をする。
5. **CI の品質ゲートを必ず通す。** マージ前に ESLint（チェック）・Prettier（整形）・`tsc`（型チェック）・テスト・`ci/policy_check.py`（禁止操作・秘密情報の検査）がすべて成功していること。
6. **正本ドキュメント（`docs/`）を常に最新に保つ。** 仕様・計画・制約を変えたら、対応する `docs/` のファイルも同じPRで更新する。
7. 作業を終えたら、**何をしたか・なぜそうしたかを、専門用語を避けた日本語で** 報告する。

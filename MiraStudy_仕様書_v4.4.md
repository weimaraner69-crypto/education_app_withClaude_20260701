## 1. 何を解決するのか（背景と方針）

### 1-1. これまでの課題
過去の試作では、子供が問題を解くたびにAI（Gemini）へ問い合わせて問題を作っていました。この方式には次の弱点がありました。

1. **品質が不安定** … AIが作った問題や答えが正しいか、人間が確認しないと分からない。
2. **速度が遅い** … 問題が出るまで待たされ、子供のやる気が下がる。
3. **図が作れない** … 文章問題ばかりで、図形やグラフの問題が作りにくい。
4. **費用がかさむ** … 問題を出すたびにAPI利用料が発生する。

### 1-2. アプリは1つに統合する（算数・数学・英語をまとめる）
英語学習も視野に入れるにあたり、「別アプリとして新しく作るか」「今のアプリに教科として追加するか」を検討し、**統合する方針**に決めました。理由は次の3つです。

1. **ログインや記録の仕組みを2重に作らなくて済む**。名前タップでのログイン、保護者の暗証番号、学習記録の保存といった土台部分は、教科が増えても1セットのままで使い回せます。
2. **もともとのデータ設計が、複数教科に対応できる形になっている**（4章・6章）。「教科」という区分けを最初から独立させてあるため、英語を追加するコストは低く抑えられます。
3. **保護者から見て、学習状況を1つの画面でまとめて確認できる**。算数と英語が別アプリだと、状況把握のために2つのアプリを行き来する必要が出てしまいます。

英語特有の機能（単語の記述・選択、文法の穴埋め・選択、リスニング）は、「アプリを分けるかどうか」の問題ではなく、**英語という教科の中に新しい問題タイプを追加する**という位置づけで扱います（6-2章）。

### 1-3. 今回の方針：「工場」と「倉庫」を分ける
問題を「使うたびに作る」のをやめ、**あらかじめ用意しておき、アプリは取り出して表示するだけ**にします。用意の仕方を2種類に分けます。

- **テンプレート方式（工場：本命）**
  「□×□＝?」のような**型（テンプレート）**を一度だけ用意し、数字はコードがその場でランダムに入れます。答えもコードが計算するので**必ず正しい**。図形やグラフも、数値に合わせてコードが描きます。**API利用料は一切かかりません。**

- **事前生成＋検証方式（倉庫：将来のAI文章問題用）**
  文章問題だけはAIに作らせますが、その場ではなく**月に一度まとめて生成**し、正しさを自動＋目視でチェックしてから保管庫に貯めます。利用時は保管庫から取り出すだけなので、速く・安くなります。

> この方針により、1-1で挙げた4つの課題（品質・速度・図・費用）がまとめて解決されます。

## 2. 目標（ゴール）とやらないこと（非ゴール）

### 2-1. ゴール
1. **問題が一瞬で出る**（体感で待たされない。目安：0.5秒以内）。
2. **答えが必ず正しい**（テンプレート方式では、コードが計算するため誤答が原理的に発生しない）。
3. **図形・グラフの問題も出せる**（コードが数値どおりにSVGで描画する）。
4. **費用を抑える**（Phase 1のテンプレート方式はAPI料金ゼロ。将来のAI問題も月次バッチで最小化）。
5. **後から機能を足せる**（AI文章問題・多教科・個別最適化を、作り直しなしで追加できる設計にする）。

### 2-2. 非ゴール（この版では作らないこと）
1. **その場でのAI問題生成**（=旧方式。速度・費用・品質の問題があるため採用しない）。
2. **Phase 1でのAI文章問題**（設計だけ用意し、実装は Phase 2 に回す）。
3. **Phase 1での英語教科の実装**（データ設計・問題タイプの枠は今回用意するが、中身の実装は算数・数学のあとに回す）。
4. **Phase 1での高校内容の実装**（学年の枠は広めに設計するが、中身はまず小6・中2の範囲から）。
5. **オンライン対戦・SNS的な機能**（家族内の学習に集中する）。

## 4. 全体の考え方（アーキテクチャ）

### 4-1. 問題の「共通の形」を最初に決める（最重要）
このアプリの土台となる考え方です。**テンプレートが作る問題も、将来AIが作る問題も、まったく同じ「形」で出てくるようにします。** 画面側は「どちらが作った問題か」を気にせず、同じように表示・採点できます。これが「後から作り直さなくて済む」秘訣です。TypeScript では、この「共通の形」を**型（`type` / `interface`）**として `src/types` に定義します。

「問題ひとつ」が持つ情報（＝共通の形）：

| 項目 | 意味 | 例 |
|------|------|----|
| source | 誰が作ったか | `template`（テンプレート）／`ai`（AI生成） |
| subjectId / gradeId / unitId | 教科・学年・単元 | 算数／小3／かけ算 |
| prompt | 問題文 | 「7 × 8 = ?」 |
| figure | 図（任意） | 三角形のSVG図。無い場合もある |
| answer | こたえ（後述の「こたえの型」） | 数字の56 など |
| choices | 選択肢（任意） | 文章問題で使う。今は空 |
| explanation | 解説（任意） | AI問題で使う。今は空 |
| **hints** | **まちがえたときに出す、3段階のヒント（配列）** | **後述4-1a参照** |
| difficulty | むずかしさ | 1〜5 |

### 4-1a. まちがえたときの「3段階ヒント」のしくみ
子供が答えをまちがえたとき、いきなり正解を見せるのではなく、**だんだん詳しくなる3段階のヒント**を出します。1回まちがえるごとに1段階ずつヒントが出て、それでも解けなければ最後に正解と解説を見せる、という流れです。

| 段階 | 呼び方 | 役割 | 例（「7 × 8 = ?」の場合） |
|------|--------|------|---------------------------|
| 1回目のまちがい | ヒント1（気づき） | 考える方向だけを示す。答えには触れない | 「7を何回たせばいいか、考えてみよう」 |
| 2回目のまちがい | ヒント2（やり方） | 解き方の手順を、より具体的に示す | 「7 × 8 は、7 × 7（＝49）に、もう1回7をたすと出るよ」 |
| 3回目のまちがい | ヒント3（ほぼ答え） | 最後の一歩だけ残す | 「49 に 7 をたすと？」 |
| 4回目もまちがい／「答えを見る」を押した | 正解表示 | 正しい答えと、簡単な解説を見せる | 「答えは 56 だよ。7 × 8 は、7を8回たした数だよ」 |

**このしくみを、テンプレート方式・AI方式の両方でどう実現するかは、次のとおりです。**

- **テンプレート方式（計算・図形・グラフ）**：ヒントも**コードが自動で作ります**。たとえば「かけ算」のテンプレートなら、「a×bの答えを、a×(b-1)の答えに a をたす形で説明する」というヒントの型を、問題の型（テンプレート）とセットで1つだけ用意すれば、数字が変わっても自動的に正しいヒントが出ます。人が毎回ヒント文を書く必要はありません。
- **AI生成方式（将来の文章問題）**：問題を事前生成するとき（7章）に、**ヒントも3段階分まとめてAIに作らせ、答えと一緒に事前チェックします**。これにより、出題時にはヒントも即座に（追加のAPI料金なしで）表示できます。

> こうしておくことで、「ヒントを出す」という機能自体は算数にも英語にも共通の仕組みとして使い回せます。英語の文法問題でも、「ヒント1：時制に注目してみよう」→「ヒント2：主語がheだから動詞の形は？」→「ヒント3：is か are、どっちかな？」のように、同じ3段階の考え方がそのまま使えます。

### 4-2. 「こたえの型」を最初から数種類そろえておく
「こたえ」の持ち方を、**今使う分＋将来使う分**まで先に用意します。これにより、AI文章問題（選択式）を足すときにデータ構造を変えずに済みます。

| こたえの型 | 中身 | いつ使うか |
|-----------|------|-----------|
| number | 数字ひとつ | 計算・図形・グラフ（**今すぐ**） |
| divmod | しょう と あまり | わり算（**今すぐ**） |
| choice | 選択肢の何番目が正解か | AI文章問題の多肢選択（**将来**） |
| text | 許容する記述の一覧 | 記述式（**さらに将来**） |

> 「今は number と divmod しか使わないのに、choice と text も定義しておく」——これが、英理究さんの指示された「初期段階から拡張できる設計にする」の具体的な中身です。

### 4-3. 出題のしくみ（工場と倉庫の使い分け）
アプリ内に「問題を1つください」と頼む窓口（＝出題担当）を1つ作ります。窓口は単元の種類を見て、自動で使い分けます。

- その単元が**テンプレート型**なら → その場でコードが問題を組み立てて返す（工場・料金ゼロ）。
- その単元が**AI生成型**なら → Firestore（保管庫）から1件取り出して返す（倉庫）。

画面側は窓口に頼むだけで、裏で工場・倉庫どちらが動いたかを知る必要がありません。この窓口は `src/lib` に共通処理として実装します。

## 5. 主な機能と優先度

優先度の意味：**P0＝これが無いと成り立たない／P1＝早めに足したい／P2＝将来**。

### P0（Phase 1で必ず作る）
- 名前タップでのログイン、保護者PINでの管理画面。
- 教科・学年・単元の選択。
- **テンプレート方式の出題・採点**（計算・図形・グラフ）。
- 正誤の判定と、はげましの表示。
- **まちがえたときの3段階ヒント表示**（4-1a参照。1回目〜3回目のまちがいで段階的にヒントを出し、それでも解けなければ正解と解説を表示）。
- 学習記録の保存（誰が・どの単元を・正解したか・何段階目のヒントで解けたか）。

### P1（Phase 2）
- **AI文章問題**（月次バッチ生成＋自動チェック＋目視承認＋保管庫出題）。
- 単元ごとの正答率・つまずき表示（保護者向け）。

### P2（Phase 3以降）
- 習熟度に応じた個別最適化（苦手を重点的に出す）。
- 多教科（国語・英語など）。
- ごほうび／継続のしくみ（連続日数、シールなど）。

## 6. データベース設計（保管庫の設計）

Firestore（Firebaseのデータ保管庫）を使います。イメージは「ラベル付きの箱（＝コレクション）」がいくつかあり、その中に「1件ずつのカード（＝ドキュメント）」が入る、という構造です。

### 6-1. 箱（コレクション）の一覧

| 箱の名前 | 入れるもの | なぜ必要か |
|----------|-----------|-----------|
| subjects | 教科（算数・数学、英語…） | 複数教科に広げるための土台 |
| grades | 学年（小1〜高3まで枠を用意） | 学年ごとに出し分ける。今使うのは小6・中2の枠のみ |
| units | 単元（かけ算、三角形の面積…） | 「どの単元の問題か」を管理 |
| problemTypes | 問題タイプ＝型の定義 | テンプレか AI かをここで区別 |
| problems | AIが作った問題そのもの | **将来の倉庫**。今は空でよい |
| users | 子供・保護者 | ログインと権限（名前タップ／PIN） |
| attempts | 解答の記録（正誤・時間） | 学習記録・分析の元データ |
| progress | 単元ごとの習熟度 | 個別最適化（Phase 3）の土台 |
| generationBatches | AI生成の実行履歴 | 費用・件数の管理（Phase 2） |

### 6-2. 各箱の中身（主な項目）

**units（単元）** ※マスターデータ。人が一度だけ用意する
- subjectId, gradeId, name（例：かけ算〈九九〉）, order（並び順）, description

**problemTypes（問題タイプ＝型）**
- unitId, name（例：九九の計算）
- **method：`template` か `generated`**（テンプレート方式か、AI生成方式か）
- generatorKey：テンプレートの場合、どのコードで作るかの目印（例：`mul`）
- **answerFormat：こたえの型**（`number`／`divmod`／`choice`／`text`／`audio` ※後述）
- difficulty：むずかしさ

**英語の問題タイプについて**

英語は算数・数学と問題の性質が異なるため、`problemTypes` の枠組みはそのまま使いつつ、英語特有の項目を追加します。想定している型は次のとおりです。

| 単元の例 | 出題形式 | こたえの型 | 備考 |
|----------|---------|-----------|------|
| 単語（記述式） | 意味や日本語を見て、英単語をつづる | `text`（複数の正解表記を許容） | 大文字・小文字、スペルミスの扱いを別途決める |
| 単語（選択式） | 4択などから正しい単語・意味を選ぶ | `choice` | テンプレート方式でも対応しやすい |
| 文法（穴埋め） | 文中の空欄に適語を入れる | `text` または `choice`（選択肢を出す場合） | 「文章問題」に近く、AI生成が向く |
| 文法（選択式） | 正しい文法を4択などから選ぶ | `choice` | テンプレート方式でも対応しやすい |
| リスニング（将来） | 音声を聞いて答える | `choice` または `text` | 音声ファイルの再生が必要。**新しい項目 `audioUrl` を problems に追加**して対応 |

> 単語・文法の「選択式」は、あらかじめ単語帳・文法パターンをテンプレートとして用意すれば、算数と同じ**テンプレート方式（費用ゼロ・即時表示）**で作れます。一方、「記述式」や「穴埋め」は自然な例文が必要になるため、**AI生成＋事前チェック方式**が向いています。リスニングは音声データの用意が必要なため、実装の難易度が最も高く、実現できるかどうかを含めてPhase 2以降に検討します（10章のOpen Questionsに追記）。

**problems（AIが作った問題）** ※Phase 2から中身が入る。形は4-1と同じ
- unitId, problemTypeId, subjectId, gradeId
- source：`ai`
- prompt, figure, answer, choices, explanation, difficulty
- **hints：3段階のヒント文（配列）** … 4-1a参照。事前生成の段階で、答え・解説と一緒にAIに作らせ、事前チェックする
- **status：`draft`（下書き）／`active`（使用可）／`rejected`（不採用）** … 品質チェックの合否を表す
- verifiedAt, verifiedBy（いつ・誰が確認したか）

**users（子供・保護者）**
- displayName, avatar（アイコン）, role：`child` か `parent`
- gradeId（子供の学年）, pinHash（保護者のみ。暗証番号は暗号化して保存）

**attempts（解答記録）**
- userId, subjectId, gradeId, unitId, problemTypeId
- （AI問題なら）problemId
- isCorrect（正解か）, answeredAt, timeTakenMs（解答にかかった時間）
- **hintsUsedCount：何段階目のヒントまで見て解けたか（0＝ヒントなしで正解）** … 保護者がつまずきの深さを把握するために記録

**progress（習熟度）**
- userId, unitId, attempts（挑戦数）, correct（正解数）, masteryLevel（習熟レベル）

### 6-3. この設計が「拡張できる」理由（まとめ）
- 問題の**共通の形**（4-1）を最初に決めてあるので、AIが作った問題も同じ棚に並ぶ。
- **こたえの型**（4-2）に選択式・記述式を先に用意してあるので、文章問題を足してもデータ構造を変えない。
- 出題は**窓口ひとつ**（4-3）を通すので、テンプレ→AIの切り替えで画面を作り直さない。
- 教科・学年を**箱として分離**しているので、中学数学や国語を足すのは「単元とテンプレを追加するだけ」。

## 7. 問題の品質を担保するしくみ

### 7-1. テンプレート方式（Phase 1）
- 答えは**コードが計算**する（例：`7 × 8` の答えはコードが掛け算する）。誤答は原理的に起きない。
- 図形・グラフは**数値どおりにコードがSVGで描画**する。数字と図がずれない。
- 計算・ヒント生成のロジックには **自動テスト（Vitest など）**を必ず添え、「7×8のテンプレートは必ず56を返すか」などを機械的に確認する。
- → 人による事前チェックは不要。

### 7-2. AI文章問題（Phase 2）
月に一度、次の流れで倉庫に貯めます。
1. Gemini に、対象単元の文章問題をまとめて生成させる。
2. 生成物を機械的にチェック（答えの再計算・形式の確認など）。おかしいものは `rejected`。
3. 英理究さんが数分、抜き打ちで目視確認し `active` にする。
4. `active` になった問題だけが子供に出題される。
- Batch API（まとめて処理すると割安になるしくみ）を使い、費用を抑える。

## 11. 開発ルール・コーディング規約（複数AIエージェント対応）

### 11-0. なぜこの章が必要か
このアプリは主に **Claude Code**（コードを書いてくれるAIツール）を使って開発しますが、**別のAIエージェント（他社のAIツールなど）が同じリポジトリ（コードの保管庫）を触る可能性**があります。

AIエージェントは、それぞれ「クセ」を持っています。たとえば、ある日はファイル名を `myComponent.tsx` のように書き、別の日は `MyComponent.tsx` のように書く、といったことが起こり得ます。人間のエンジニアチームでも、こうした「書き方のばらつき」は事前にルールを決めておかないと発生します。**このルールが無いと、後から読み返したときに「誰が・どんな考えでこの部分を書いたのか」が分からなくなり、修正が難しくなる**という問題が起きます。この章は、それを防ぐための取り決めです。

### 11-1. 基本方針：Claude Codeの標準的な書き方＋テンプレートの進め方をベースにする
特別な理由がない限り、**Claude Codeが標準で採用する書き方（下記11-2）を「正」とします。** また、開発の“進め方”は `dev-orchestration-template`（言語非依存のひな形）の「計画→実装→三重監査→リリース」ワークフローに沿います。他のAIエージェントに作業を頼む場合も、このリポジトリの中では同じ書き方・進め方に合わせてもらいます。

これを実現するために、リポジトリの一番上の階層に **`CLAUDE.md`** というファイルを置きます。これは「このプロジェクトのルールブック」にあたるファイルで、Claude Codeは自動的にこのファイルを読み込んで従います。**他社のAIエージェントを使う場合は、作業を頼む最初のメッセージで「まず CLAUDE.md を読んでから作業してください」と一言添える**ことで、同じルールに揃えられます。より詳しい進め方・制約は `docs/`（正本ドキュメント）にまとまっています。

### 11-2. コードの書き方（規約の中身）

| 項目 | ルール | 補足（なぜこう決めるか） |
|------|--------|--------------------------|
| 使用言語 | **TypeScript**（JavaScriptに「型」という安全装置を足した言語） | 型があると、AIが書いたコードに矛盾があった場合、実行する前にエラーとして検出できる。複数のAIが触るときほど、この安全装置が重要になる |
| コンポーネントの書き方 | Reactの「関数コンポーネント」＋ Hooks（`useState` など）で統一。クラス形式は使わない | Claude Codeが標準で採用する書き方。統一しないとファイルごとに作法が変わってしまう |
| ファイル名 | 画面部品（コンポーネント）は `PascalCase.tsx`（例：`QuestionCard.tsx`）。それ以外（関数・設定など）は `camelCase.ts`（例：`generateProblem.ts`） | Claude Codeの標準的な命名慣習 |
| 変数・関数名 | `camelCase`（例：`currentUnit`）。定数は `UPPER_SNAKE_CASE`（例：`MAX_HINT_LEVEL`） | 同上 |
| フォルダ構成 | `src/components`（画面部品）／`src/features`（機能ごとのまとまり）／`src/lib`（共通処理）／`src/types`（型の定義） | 「どこに何があるか」を、AIが変わっても迷わない配置にする |
| コメント | 複雑な処理（テンプレートのロジック、ヒント生成など）には、**日本語で「なぜこう書いたか」を一言添える** | 英理究さん自身がコードを読むときの手がかりにもなる |
| フォーマット（見た目の整形） | **Prettier**（自動整形ツール）と **ESLint**（自動チェックツール）の設定ファイルをリポジトリに含める | 「タブか半角スペースか」のような見た目の差異を、ツールが自動で揃えてくれる。AIごとの表記ゆれを機械的になくせる |
| コミットメッセージ | **Conventional Commits** 形式：`feat: 単元選択画面を追加`、`fix: ヒントの計算が1個ずれる不具合を修正` のように、種類（feat／fix／docs／refactor／test／chore）を頭につける | どのAIが・いつ・何を変えたかを、あとから履歴だけで追える |
| テスト | 答えの計算ロジック（テンプレート）には、**簡単な自動テスト**（Vitest など）を必ず添える（例：「7×8のテンプレートは、必ず56を返すか」を確認するテスト） | 4-1a・7-1章の「答えは必ず正しい」という保証を、コードの面からも裏付ける |

### 11-3. 複数のAIエージェントが作業する場合の運用ルール（重要）

これは「誰が書いたか分からないコードが紛れ込む」事故を防ぐための、作業の進め方のルールです。`dev-orchestration-template` の「計画→実装→三重監査→リリース」ワークフローに沿って進めます。**特に、Claude Code以外のAIエージェントに作業を頼むときは、必ずこの手順を守ってください。**

1. **作業前に、必ず `CLAUDE.md` を読ませる。** 「このルールに沿って書いてください」と最初に伝える。
2. **メインの完成版（`main`ブランチ）に、直接書き込ませない。** 必ず作業用の「枝分かれ」（＝ブランチ）を新しく作らせてから、そこで作業させる。これにより、うまくいかなかった場合に元に戻しやすくなる。
3. **実装したら、テスト・三重監査（仕様・セキュリティ・信頼性）を通す。** ひな形のエージェント構成（`agents/`・`.github/agents/`）に沿って、「必ず直すべき指摘」がゼロになるまで直す。
4. **CIの品質ゲートを必ず通す。** マージ前に ESLint・Prettier・`tsc`（型チェック）・テスト・`ci/policy_check.py`（禁止操作・秘密情報の検査）がすべて成功していること。
5. **11-2の規約からズレたコードが出てきたら、機械的に上書きしない。** まず**差分（さぶん＝「変更前と変更後で、どこがどう変わったか」の一覧）を確認**し、なぜそう書いたのか、意図を確認する。単なる書きぐせのズレなら11-2に合わせて直すが、**意図がある変更（新しい考え方の提案など）の場合は、英理究さんに確認してから採用・不採用を決める。**
6. **差分は必ず記録として残す。** 「なんとなく直した」ではなく、**「変更前・変更後・変更理由」をコミット履歴として保存**する。これにより、後から「なぜこうなったか」を誰でも追跡できる。
7. **リポジトリ全体に影響する大きな変更（データ構造、命名規則の変更など）は、直接書き込まず、必ず PR（プルリクエスト）を作る。** PRとは「この変更を取り込んでよいか、正式に確認・レビューするための提案書」のようなものです。英理究さんが内容を確認し、問題なければ「取り込む（マージする）」操作をして初めてメインの完成版に反映されます。**判断に迷う変更・大きな変更は、必ずこの一段階を挟む**ことをルールとします。

> かんたんに言うと：**「勝手に本番へ反映しない」「変えたところは必ず記録に残す」「大きい変更は一度提案の形で止めて、確認してから取り込む」「実装したら必ずテストと監査を通す」**が徹底されていれば、AIが複数人（複数体？）入れ替わり立ち替わり作業しても、リポジトリが壊れることはありません。

### 11-4. この章の運用開始タイミング
この規約は、**PLAN.mdのタスク1-3「開発環境の準備」の中で、`CLAUDE.md` ファイルと Prettier・ESLint（自動整形・自動チェックツール）の設定、および `dev-orchestration-template` のブートストラップ（`scripts/bootstrap.sh`）を最初に用意する**ことで、実際の開発に組み込みます。土台の一番最初に仕込んでおくことで、後から「規約を守っていないコードだらけになってから慌てて直す」事態を防ぎます。

## 付録：用語集（プログラミング用語のやさしい説明）

- **dev-orchestration-template**：このプロジェクトの“進め方”の土台にしている「ひな形」。開発の流儀（計画→実装→三重監査→リリース）、CI（自動チェック）、AIエージェント構成が入っている。言語非依存で、アプリ本体は React / TypeScript で作る。
- **三重監査**：実装したコードを「仕様どおりか」「安全か」「壊れにくいか」の3つの視点で独立にチェックすること。dev-orchestration-template のエージェント構成が担う。
- **CI（シーアイ）**：コードを変えるたびに、ESLint・Prettier・`tsc`・テストなどを自動で実行して品質を確かめるしくみ。
- **Firebase / Firestore**：Googleが提供する、データを保管しておく場所（クラウド上の保管庫）とその機能一式。サーバーを自前で用意しなくても使える。
- **コレクション／ドキュメント**：Firestoreの構造。コレクション＝「ラベル付きの箱」、ドキュメント＝「箱に入った1件のカード」。
- **テンプレート（型）**：問題の「ひな形」。数字を差し替えるだけで無限に問題が作れる。
- **API（エーピーアイ）**：外部のサービス（今回はAI）に処理を頼むための窓口。使うたびに料金が発生することがある。
- **バッチ処理**：たくさんの処理をまとめて一度に実行すること。都度実行より速く・安くなりやすい。
- **SVG（エスブイジー）**：図形を数値で表す描画のしくみ。コードで正確な図が描け、拡大してもきれいなまま。
- **PIN**：暗証番号。ここでは保護者が管理画面に入るための4桁の番号。
- **P0 / P1 / P2**：優先度。P0＝必須、P1＝早めに、P2＝将来。
- **リポジトリ**：コード一式が入った保管庫（フォルダのようなもの）。バージョン管理ツール「Git」で変更履歴が記録される。
- **ブランチ**：リポジトリの中で、本番（main）とは別に作業できる「枝分かれ」。安全に試作・修正ができる。
- **コミット**：変更内容を記録として保存する操作。「いつ・何を・なぜ変えたか」の履歴になる。
- **差分（diff）**：変更前と変更後で、どこがどう変わったかの一覧。
- **PR（プルリクエスト）**：ブランチでの変更を「本番に取り込んでよいか」提案し、確認・レビューしてから取り込むためのしくみ。
- **CLAUDE.md**：Claude Codeが自動的に読み込む、プロジェクトごとのルールブックにあたるファイル。
- **TypeScript**：JavaScriptに「型」という安全装置を加えたプログラミング言語。コードの矛盾を実行前に検出しやすくなる。
- **Prettier / ESLint**：コードの見た目を自動で整えたり、書き方の間違いを自動チェックしたりするツール。
